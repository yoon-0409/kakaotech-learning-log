# VanillaJS Todo App

`HTML / CSS / Vanilla JS`만으로 구현한 날짜 기반 Todo 웹 앱.
라이브러리·프레임워크 없이 DOM 조작과 이벤트 핸들링을 직접 다뤘다.

## 실행 방법

VSCode의 **Live Server** 확장으로 `index.html`을 실행한다.
(`file://`로 직접 열면 일부 동작이 제한될 수 있어 Live Server 사용을 권장한다.)

## 파일 구조

```
todo-vanilla/
├── index.html
├── style.css
├── app.js
└── README.md
```

## 구현 기능

### 기본 기능

- **CRUD**: 할 일 생성 / 조회 / 완료 토글 / 삭제
- **입력 검증**: 빈 값으로 추가 시 토스트 알림으로 안내
- **상태별 필터링**: 전체 / 진행중 / 완료 탭으로 분류
- **일간 뷰**: 선택된 날짜의 할 일만 표시, 이전·다음 버튼으로 날짜 이동
- **달력 선택**: `input type="date"`로 특정 날짜 직접 선택
- **로컬스토리지 연동**: 변경 시마다 저장, 새로고침 후에도 데이터 유지

### 추가 구현

- **전체 조회 모드**: 날짜와 무관하게 모든 할 일을 한 번에 표시 (각 항목에 날짜 함께 표기)
- **빈 상태 처리**: 해당 날짜에 할 일이 없을 때 안내 표시

## 핵심 설계

데이터(배열)는 한 곳에만 두고, 화면은 그 데이터를 매번 다시 그리는 방식으로 구성했다.

```
todo 배열 (데이터)
   → 현재 필터 / 선택된 날짜 기준으로 걸러서
   → renderTodoList()로 목록을 다시 그림
```

상태가 바뀌면 데이터만 수정하고 `renderTodoList()`를 다시 호출한다.
날짜 비교는 `Date` 객체를 직접 비교하지 않고 `"YYYY-MM-DD"` 문자열(`toDateKey`)로 통일해 혼란을 줄였다.

## 트러블슈팅

### 1. `Cannot read properties of null (reading 'addEventListener')`

`script` 태그를 `<head>`의 `link` 아래에 습관적으로 작성한 것이 원인이었다.
JS가 실행되는 시점에 아직 DOM 요소가 생성되지 않아 `getElementById`가 `null`을 반환했다.
예전에도 겪었던 문제인데 무심코 또 같은 위치에 적었다. `script`를 `</body>` 직전으로 옮겨 해결했다.

### 2. active 필터 버튼의 배경색만 적용되지 않음

글자색(`color`)은 적용되는데 배경색(`background`)만 먹지 않는 증상이었다.
원인은 CSS 명시도(specificity) 충돌이었다. `:not(#add-btn)` 셀렉터에 id가 포함되어
명시도가 높아진 탓에, 클래스만으로 지정한 `.active`의 배경색을 덮어쓰고 있었다.
같은 규칙 안에 없던 `color`만 정상 적용된 것이 단서였다.
id가 끼지 않도록 `.filter-btn` 클래스 기반 셀렉터로 정리해 해결했다.

## 소감

이번 과제는 AI 도움을 최소화하고, CSS만 AI로 작성해 디자인에 대한 부담을 덜었다.
대부분의 코드를 직접 타이핑했는데, 오랜만에 다뤄보는 바닐라 JS라 재미있기도 했고 복습도 됐다.

JS가 매 순간 로드되어 버튼의 클릭 이벤트를 조회하고, 코드가 실시간으로 동작하고 있다는 점은
여전히 신기하게 느껴진다. JS만의 재미있는 지점이라고 생각한다.

AI가 짜준 코드를 복사·붙여넣기 하면 흐름을 이해하는 데 어려움이 많은데,
직접 타이핑하니 코드 흐름이 훨씬 명확하게 눈에 들어오고 이해도 잘 됐으며, 의구심도 자연스럽게 생겼다.
내가 직접 쓰고 통제할 수 있는 코드라야 비로소 흐름이 보인다는 감각.
어쩌면 이래서 내가 Unix 스타일을 선호하는지도 모른다.
작고 단순하며, 내가 이해하고 통제할 수 있는 도구를 조합하는 방식 — 그 통제 가능한 명료함이
"마법처럼 알아서 돌아가는" 코드보다 나에게 잘 맞는다.
